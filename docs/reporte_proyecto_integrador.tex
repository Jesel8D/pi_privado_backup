\section{Resumen}
TienditaCampus es un sistema web para gestionar inventarios y ventas, diseñado para estudiantes que venden comida dentro de universidades. El sistema usa el \textbf{Método IQR} (Rango Intercuartílico) para analizar el historial de ventas del estudiante y sugerirle cuánto producto debe comprar, evitando que compre de más y se le eche a perder.

La aplicación fue construida con \textbf{Next.js 14} en el frontend, \textbf{NestJS} en el backend, \textbf{PostgreSQL} como base de datos, y todo empaquetado con \textbf{Docker}. Además, se conectó a \textbf{Google BigQuery} para auditoría en la nube.

% 3. INTRODUCCIÓN

\section{Introducción}
En las universidades mexicanas, muchos estudiantes venden sándwiches, postres y botanas para pagar sus gastos. El problema es que llevan sus cuentas ``al tanteo'': anotan en libretas o simplemente recuerdan cuánto compraron y cuánto vendieron. Esto hace que compren de más (y la comida se echa a perder) o compren de menos (y pierdan ventas).

TienditaCampus nace para resolver esto. Es una plataforma web que le permite al estudiante registrar sus productos, llevar el control de su inventario, procesar ventas desde el celular y, lo más importante, recibir sugerencias basadas en estadística sobre cuánto debe comprar la próxima vez. Todo usando tecnologías modernas como React, Node.js y Google Cloud, cumpliendo con las normas de higiene alimentaria y protección de datos de México.

\section{Planteamiento del problema}

\subsection{Descripción del problema en su contexto}
En la Universidad Politécnica de Chiapas (y en universidades en general) hay estudiantes que venden comida todos los días. Su problema principal es que no saben cuánto producto comprar. Llevan sus registros de forma manual, en papel o de memoria, lo que causa una mala rotación de inventario.

\subsection{Antecedentes y evidencia}
México desperdicia aproximadamente 30 millones de toneladas de alimentos al año. A nivel universitario, cuando un estudiante compra mucho producto sin basarse en datos reales de sus ventas pasadas, termina tirando lo que no vendió. Las aplicaciones que ya existen no le sirven: los Puntos de Venta como \textit{Square} o \textit{Toast} cobran comisiones altísimas que un estudiante no puede pagar. Las apps de rescate de comida como \textit{Too Good To Go} o \textit{Infood} solo ayudan a vender las sobras baratas, pero no previenen el desperdicio desde el origen.

\subsection{Declaración clara del problema}
No existen herramientas accesibles que analicen el historial de ventas de un estudiante vendedor y le digan cuánto producto debe comprar. Esto causa desperdicio de comida y pérdida de dinero.

\subsection{Consecuencias del problema}
\begin{enumerate}
    \item Pérdida del dinero invertido en producto que se echó a perder.
    \item Aumento del desperdicio de alimentos dentro del campus.
    \item Riesgo sanitario al no saber qué producto entró primero al inventario.
\end{enumerate}

\subsection{Delimitación del problema}
El sistema se limita a ser una aplicación web responsive (funciona en celular y computadora). No incluye logística de reparto fuera del campus ni pagos con tarjeta. El algoritmo de predicción usa exclusivamente el método IQR.

\subsection{Preguntas de investigación}
\begin{itemize}
    \item ¿El uso del Método IQR para filtrar datos de ventas puede reducir el desperdicio de producto en los vendedores universitarios?
    \item ¿Una arquitectura Monorepo con Docker mejora la mantenibilidad del sistema a largo plazo?
\end{itemize}

\subsection{Justificación del proyecto}
El proyecto se justifica en dos sentidos: socialmente, protege el dinero del estudiante emprendedor y reduce el desperdicio; académicamente, permite al equipo aplicar bases de datos avanzadas (PostgreSQL con extensiones), arquitecturas de software modernas (NestJS, Docker) y conexión a la nube (BigQuery).

\section{Marco teórico}

\subsection{Antecedentes teóricos y tecnológicos}
Los Puntos de Venta (POS) tradicionales fueron hechos para negocios grandes y establecidos. Para el nicho universitario, existen algunas alternativas pero ninguna resuelve el problema de fondo:

\begin{itemize}
    \item \textbf{Square y Toast:} Son terminales de pago que cobran comisiones del 3\% o más por transacción. Un estudiante que vende tortas de \$25 pesos no puede absorber ese costo.
    \item \textbf{FáciLUNCH e Infood:} Son apps universitarias de pedidos y entrega. Ayudan a distribuir la comida, pero no ayudan al vendedor a saber cuánto producto comprar.
    \item \textbf{Too Good To Go:} Vende sobras de comida a precio rebajado. Es una solución reactiva (después de que ya se echó a perder), no preventiva.
\end{itemize}

TienditaCampus es diferente porque ataca el problema antes de que suceda: analiza las ventas pasadas y sugiere cuánto comprar.

\subsection{Conceptos clave}
\begin{itemize}
    \item \textbf{Merma:} Producto que se pierde o se echa a perder. En nuestro contexto, comida que caduca sin venderse.
    \item \textbf{FIFO (First In, First Out):} Método de rotación de inventario donde lo primero que entra al almacén es lo primero que debe salir. Evita que productos viejos se queden al fondo.
    \item \textbf{Monorepo:} Guardar todo el código del proyecto (frontend, backend, configuración) en un solo repositorio de Git.
    \item \textbf{SSR (Server Side Rendering):} Técnica donde el servidor genera la página HTML completa antes de enviarla al navegador. Hace que la aplicación cargue rápido incluso con internet malo.
    \item \textbf{JWT (JSON Web Token):} Un token cifrado que se usa para verificar la identidad del usuario en cada petición al servidor.
    \item \textbf{Docker:} Herramienta que empaqueta la aplicación y todas sus dependencias en un contenedor aislado, garantizando que funcione igual en cualquier computadora.
\end{itemize}

\subsection{Tecnologías utilizadas}

\subsubsection{Frontend: Next.js 14 con React}
Next.js es un framework de React que permite renderizar las páginas desde el servidor (SSR). Esto es clave porque los estudiantes acceden desde sus celulares con el Wi-Fi del campus, que a veces es lento. Con SSR, la página llega lista para verse sin esperar a que JavaScript cargue todo.

\subsubsection{Backend: NestJS con TypeScript}
NestJS es un framework para Node.js que organiza el código en módulos, controladores y servicios. El uso de TypeScript (en lugar de JavaScript puro) nos permite detectar errores antes de ejecutar el código, algo importante cuando se hacen cálculos matemáticos como el IQR.

\subsubsection{Base de Datos: PostgreSQL 16}
PostgreSQL es una base de datos relacional de código abierto. Lo elegimos por su estabilidad, su soporte para transacciones complejas y porque tiene extensiones como \texttt{pg\_stat\_statements} que nos permiten medir el rendimiento de cada consulta SQL.

\subsubsection{Contenedores: Docker Compose}
Docker Compose nos permite levantar todo el sistema (frontend, backend y base de datos) con un solo comando. Cada servicio corre en su propio contenedor aislado. Si la base de datos se reinicia, los datos se preservan porque están en un volumen persistente.

\subsubsection{Nube: Google BigQuery}
BigQuery es un servicio de Google Cloud para analizar grandes cantidades de datos. Lo usamos para exportar métricas de rendimiento de la base de datos. Esto permite que un profesor o auditor revise en tiempo real cuántas consultas se hicieron y cuántos recursos consumieron.

\subsection{El Método IQR (Rango Intercuartílico)}
Este es el corazón matemático de TienditaCampus. Cuando un estudiante registra sus ventas diarias, algunos días puede tener picos irreales (por ejemplo, vendió 40 sándwiches un día de evento cuando normalmente vende 10). Si simplemente sacamos el promedio, nos daría un número inflado y el estudiante compraría de más.

El IQR resuelve esto eliminando esos valores extremos (outliers). Funciona así:

\begin{enumerate}
    \item Se ordenan todas las ventas de menor a mayor.
    \item Se calcula el Primer Cuartil ($Q_1$): el valor donde se acumula el 25\% de los datos.
    \item Se calcula el Tercer Cuartil ($Q_3$): el valor donde se acumula el 75\% de los datos.
    \item Se obtiene el IQR:
\end{enumerate}

\begin{equation}
IQR = Q_3 - Q_1
\end{equation}

Con ese valor se establecen límites:
\begin{equation}
L\acute{\imath}mite_{Inferior} = Q_1 - 1.5 \times IQR
\end{equation}
\begin{equation}
L\acute{\imath}mite_{Superior} = Q_3 + 1.5 \times IQR
\end{equation}

Cualquier venta que caiga fuera de esos límites se ignora para el cálculo de reabastecimiento. Así, el sistema le sugiere al vendedor un número realista de cuánto producto comprar.

\section{Normativas}
El sistema cumple con dos normas mexicanas:

\begin{itemize}
    \item \textbf{NOM-251-SSA1-2009 (Higiene Alimentaria):} El sistema obliga a despachar los productos en orden FIFO. Los productos que fueron registrados primero son los que se venden primero. Esto evita que comida vieja se quede olvidada en el inventario.
    \item \textbf{LFPDPPP (Protección de Datos Personales):} El sistema no guarda contraseñas. Los usuarios inician sesión con su cuenta de Google (SSO), así toda la seguridad la maneja Google y nosotros nunca almacenamos datos sensibles.
\end{itemize}

\end{itemize}

\section{Implementación}
El sistema se despliega con un solo comando (\texttt{docker-compose up}) que levanta tres contenedores:
\begin{enumerate}
    \item \textbf{Contenedor de PostgreSQL:} La base de datos con todas las tablas y la extensión \texttt{pg\_stat\_statements}.
    \item \textbf{Contenedor del Backend:} El servidor NestJS que expone la API REST.
    \item \textbf{Contenedor del Frontend:} La aplicación Next.js que sirve las páginas con SSR.
\end{enumerate}

El ORM utilizado es \textbf{TypeORM}, que traduce las clases de TypeScript directamente a tablas de PostgreSQL. El frontend usa un tema oscuro profesional creado con Tailwind CSS, optimizado para verse bien en pantallas de celular.

El código fuente completo se encuentra en los siguientes repositorios de GitHub:
\begin{itemize}
    \item \textbf{Repositorio principal:} \url{https://github.com/Jesel8D/pi_privado_backup}
\end{itemize}

% 12. PRUEBAS
% =========================================================
\section{Pruebas}
Durante el desarrollo se encontraron y resolvieron varios problemas técnicos:

\begin{itemize}
    \item \textbf{Error 500 en creación de productos:} La columna \texttt{seller\_id} no se estaba enviando correctamente desde el frontend. Se resolvió ajustando el servicio de autenticación para inyectar el ID del usuario.
    \item \textbf{Error 500 en BigQuery:} La extensión \texttt{pg\_stat\_statements} no estaba disponible en el contenedor de PostgreSQL. Se resolvió agregando la instrucción de carga en el \texttt{docker-compose.yml}.
    \item \textbf{Vista SQL faltante:} La vista \texttt{v\_daily\_export} no existía en la base de datos. Se resolvió creándola en el script de inicialización del contenedor.
\end{itemize}

Todas las rutas de la API fueron probadas con Postman, verificando tanto los casos exitosos como los de error (producto no encontrado, usuario no autenticado, stock insuficiente).

\section{Resultados}
Se logró un prototipo completamente funcional con las siguientes capacidades:
\begin{itemize}
    \item Registro e inicio de sesión con Google SSO.
    \item Creación y gestión completa de productos (con soporte para perecederos).
    \item Procesamiento de pedidos con cálculo automático de totales.
    \item Dashboard con resumen de ventas diarias, inversión y ganancia neta.
    \item Exportación de métricas de rendimiento SQL hacia Google BigQuery.
    \item Interfaz responsive que funciona correctamente en celulares.
\end{itemize}

El algoritmo IQR demostró ser viable para filtrar valores atípicos en los historiales de venta, generando sugerencias de compra más precisas que un simple promedio.

\section{Conclusiones}
TienditaCampus demostró que es posible construir un sistema de gestión de inventarios profesional usando tecnologías modernas, accesible para estudiantes y con un motor estadístico que realmente ayuda a reducir el desperdicio. El trabajo en equipo bajo una arquitectura Monorepo con Docker, PostgreSQL y NestJS nos permitió aplicar de forma práctica los conocimientos adquiridos en la carrera de Ingeniería en Tecnologías de la Información.

El proyecto va más allá de una simple libreta digital: es un sistema completo con autenticación segura, base de datos relacional, auditoría en la nube y análisis estadístico, todo corriendo en contenedores que se despliegan con un solo comando.
